## Executive Summary

Workflow-related tickets represent **22% of all Pylon volume** over the last 90 days. After filtering automated noise (Stripe billing alerts, auto-reply emails), **345 real customer workflow interactions** were analyzed across the full date range.

**Three dominant themes emerged:**

1. **Bugs dominate at 126 tickets (33%)** — workflow not firing, HubSpot integration failures, trigger misfires, and variable/model errors are the top categories. This is the single largest source of workflow support load and likely the biggest detractor from product confidence.
2. **High demand for net-new automation at 91 tickets (24%)** — customers are actively trying to build new workflows across CRM updates, post-call notes, follow-up emails, sales handoffs, and coaching. This represents significant expansion potential if the builder is easier to use.
3. **Configuration complexity at 56 questions (24%)** — a near-equal share of tickets are customers asking for help setting up workflows, indicating the builder UX has a self-service gap. Many of these could be resolved with better docs, templates, or guided setup.

**HubSpot is the dominant integration surface** appearing in 86 tickets, followed by Email (51) and Slack (41). CRM field updates and post-call notes are the top use cases by volume.

## Recommended Next Steps (PM Actions)

Ordered by signal strength and strategic alignment:

### 1. Trigger Reliability (P0)
**126 bug tickets, majority tied to trigger failures.** Auto-trigger post-meeting is the core value proposition. Every missed fire is a trust failure. Needs instrumentation to detect silent failures, alerting when a trigger is expected but not fired, and an audit log customers can view.

### 2. HubSpot Integration Hardening (P0)
**86 tickets touch HubSpot** — field writes, note creation, deal updates, activity logs. Many of these are bugs. HubSpot is the #1 integration and needs a dedicated reliability sprint: retry logic, better error messages, and a connection health indicator customers can see.

### 3. Workflow Visibility & Permissions (P1)
**12 visibility bugs + 9 access permission tickets = 21 combined.** The public/private/team permission model is creating confusion and support load. Customers are discovering their workflows are missing from the UI, or that team members can't access them. This is both a UX and reliability problem.

### 4. Self-Service Setup UX (P1)
**89 question tickets (24%)** are customers asking how to set up workflows they should be able to build independently. Templates, a guided setup wizard, and contextual docs in the builder would directly reduce this load. This is the biggest CS time sink that isn't a true product bug.

### 5. Duplicate / Clone Workflow (P2)
**8+ explicit requests** for cloning workflows. Teams building the same workflow for multiple accounts/reps have no way to duplicate. This is a high-frequency power user need that's currently creating disproportionate support volume.

### 6. Bulk Historical Run (P2)
**6+ requests** for running workflows on past meetings. Customers want to backfill CRM data when they adopt a new workflow. This is a common onboarding friction point and expansion unlock.

### 7. Undo / Version History (P3)
**5 tickets** about losing workflow changes with no recovery. Destructive saves create anxiety and support calls. Even a simple "restore last saved version" would eliminate this.
