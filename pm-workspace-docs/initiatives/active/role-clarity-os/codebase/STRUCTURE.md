## Directory Purposes

**`functions/src/contexts/`:**
- Purpose: Domain-driven business logic organized by feature/entity
- Contains: Context, repository, resolvers, schema, types, tests for each domain
- Structure: One subdirectory per domain (e.g., `engagements/`, `artifacts/`, `workflows/`)
- Subcontexts: Domains with cohesive sub-features use subdirectories (e.g., `workflows/workflow-assistant-runs/`)
- Naming: Plural form for directories (`engagements`, `artifacts`), singular for files (`engagement.context.ts`)

**`functions/src/db/`:**
- Purpose: Database schema, migrations, and connection utilities
- Key files:
  - `schema.ts`: Drizzle table definitions, enums, relations (source of truth)
  - `seed.ts`: Database seeding for development and testing
  - `pooled-client.ts`: Connection pool for production
  - `local-client.ts`: Direct connection for local Firebase emulator

**`functions/src/schema/`:**
- Purpose: GraphQL schema definitions
- Contains: `.graphql` files for types, queries, mutations, subscriptions
- Organization: One schema per context domain
- Generated: Types are auto-generated by graphql-codegen into `types.generated.ts`

**`functions/src/types/`:**
- Purpose: Shared TypeScript types and enums
- Contains: Application enums, union types, event payloads
- Examples: `elephant.ts` (main type definitions), `event-payloads.ts` (Pub/Sub event shapes)

**`functions/src/utils/`:**
- Purpose: Cross-cutting utilities
- Contains:
  - `logger.ts`: Logging setup with structured metadata
  - `loggingContext.ts`: Request context propagation
  - `indexer/`: Semantic search index builder and query tool
  - `cache.ts`: Valkey/Redis caching helpers
  - Type utilities and common helpers

**`apps/web/src/`:**
- Purpose: React SPA codebase (shared with Electron desktop)
- Structure:
  - `components/`: Feature components (domain-organized) + UI primitives
  - `routes/`: Page components organized by URL path
  - `hooks/`: Custom React hooks (domain-organized subfolders when needed)
  - `contexts/`: React Context providers
  - `providers/`: Apollo, Auth, and other application providers
  - `lib/gql/`: Generated GraphQL types and utility functions
  - `lib/queries/`: Shared GraphQL query definitions
  - `utils/`: Helper functions, type guards, formatters
  - `types/`: Type definitions for web layer

**`apps/web/electron/`:**
- Purpose: Electron desktop application
- Structure:
  - `main.ts`: Main process (window management, IPC handlers)
  - `preload.ts`: Preload script (IPC bridge to renderer)
  - `src/`: Renderer process components (React)
  - `ipc/`: Typed IPC channel definitions
  - `utils/`: Recording service, file system utilities
  - `hooks/`: React hooks specific to Electron (recording state, etc.)

**`apps/web/e2e/`:**
- Purpose: End-to-end tests (Playwright)
- Tests: Full user workflows across web and Electron

**`apps/web/public/`:**
- Purpose: Static assets served by Vite
- Contains: Firebase messaging service worker, email assets, favicons

## Naming Conventions

**Files:**

| Pattern | Example | Purpose |
|---------|---------|---------|
| `{model}.context.ts` | `engagement.context.ts` | Business logic for a domain |
| `{model}.repository.ts` | `engagement.repository.ts` | Database access for a domain |
| `{model}.data-source.ts` | `engagement.data-source.ts` | GraphQL data source (if used) |
| `{model}s.test.factory.ts` | `engagements.test.factory.ts` | Test fixtures for a domain |
| `{Model}.ts` (in resolvers) | `Engagement.ts` | GraphQL resolver for a type |
| `{name}.functions.ts` | `process-engagement.functions.ts` | Cloud Functions (HTTP/Pub/Sub handlers) |
| `{name}.schema.ts` (or `.graphql`) | `engagement.schema.graphql` | GraphQL type/query/mutation definitions |
| `schema.model.ts` | `schema.model.ts` | Internal TypeScript models (if separate from DB) |
| `{name}.component.tsx` | `EngagementCard.tsx` | React component |
| `{name}.hook.ts` | `useEngagementData.ts` | Custom React hook |
| `{name}.test.ts` / `.test.tsx` | `utils.test.ts` | Test files (same name as source) |

**Directories:**

| Pattern | Example | Purpose |
|---------|---------|---------|
| Plural model name | `engagements/`, `artifacts/` | Domain context directory |
| Feature subdirs | `components/engagements/`, `contexts/workflows/` | Feature-scoped collections |
| Utility categories | `utils/indexer/`, `lib/analytics/` | Utility groupings |
| Route paths | `routes/workspaces/`, `routes/admin/` | URL-aligned pages |

## Where to Add New Code

**New Feature (Complete Domain):**

1. **Backend context:** `functions/src/contexts/{feature-plural}/`
   - `{feature}.context.ts` - Orchestration logic
   - `{feature}.repository.ts` - Database access
   - `resolvers/` - GraphQL resolver files
   - `schema.graphql` - GraphQL schema
   - `schema.model.ts` - Internal types (if needed)
   - `{feature}s.test.factory.ts` - Test fixtures
   - `*.test.ts` - Test files

2. **GraphQL schema:** `functions/src/schema/{feature}.graphql`
   - Define Query/Mutation/Subscription additions
   - Run `pnpm generate:graphql` to auto-generate resolver types

3. **Database schema:** `functions/src/db/schema.ts`
   - Add table definitions
   - Run `pnpm db:generate` to auto-create migration
   - Run `pnpm db:migrate` to apply migration

4. **Web UI:** `apps/web/src/components/{feature}/`
   - Feature-specific components
   - Colocated GraphQL queries/mutations/fragments
   - Tests alongside components

**New Component/Module:**

- **UI component:** `apps/web/src/components/{feature}/{ComponentName}.tsx`
- **Custom hook:** `apps/web/src/hooks/{feature}/use{HookName}.ts`
- **Utility helper:** `apps/web/src/utils/{domain}/{functionName}.ts` or `functions/src/utils/{domain}/`
- **Page/route:** `apps/web/src/routes/{path}/{PageName}.tsx`

**New GraphQL Query/Mutation:**

1. Write in component file or shared query file: `apps/web/src/lib/queries/` or colocated in component
2. Use `graphql` from `@/lib/gql` (never `gql` from `@apollo/client`)
3. Define fragments near consumers using `useFragment()` hook
4. Spread fragments in parent queries

**New Event Handler:**

1. Create `functions/src/contexts/{feature}/{feature}.functions.ts` with handler export
2. Wrap handler in `runWithLoggingContext` with requestId, functionName, workspaceId
3. Call context function for business logic
4. Register topic subscription in Cloud Functions deployment (Pulumi or manual setup)

**New Scheduled Task:**

1. Add handler to `functions/src/contexts/scheduled-handlers/{task-name}.ts`
2. Export async function matching Cloud Scheduler signature
3. Wrap in logging context, iterate workspaces, call context functions
4. Register in Cloud Scheduler configuration

## Special Directories

**`functions/drizzle/`:**
- Purpose: Auto-generated database migrations (do not edit manually)
- Generated: By running `pnpm db:generate`
- Committed: Yes, track migrations in git
- Structure: Timestamped migration files with SQL

**`functions/lib/`:**
- Purpose: Compiled output from `functions/src/` (do not edit)
- Generated: By `pnpm -F functions build` or watch mode
- Committed: No, output directory only

**`apps/web/dist/`:**
- Purpose: Compiled web SPA (Vite output)
- Generated: By `pnpm -F web build`
- Committed: No, build artifact only

**`apps/web/.next/`:**
- Purpose: Vite/Next output cache
- Committed: No, cache directory

**`node_modules/` and workspace `node_modules/`:**
- Purpose: Installed dependencies
- Committed: No, lock file (`pnpm-lock.yaml`) is committed instead
- Install: `pnpm install` at root

**`emulators-snapshot/`:**
- Purpose: Firebase emulator state (Firestore, Storage, Auth) captured for quick local startup
- Committed: Yes, checked in so developers start with consistent data
- Clear: Run `pnpm start:services:refreshdb` to start fresh

**`.planning/codebase/`:**
- Purpose: Architecture and structure analysis documents (this directory)
- Committed: Yes, reference for future development
